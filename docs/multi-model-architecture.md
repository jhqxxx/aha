## 数据流示例

### 场景：同时处理文本和语音请求

```
时间线:
T0: 用户A发送文本请求 → /v1/chat/completions (model: Qwen3)
T1: 用户B发送语音请求 → /v1/audio/transcriptions (model: GlmAsrNano)

执行流程:
┌─────────────────────────────────────────────────────────┐
│ T0: Chat Handler                                         │
│ 1. 解析请求,提取 model="Qwen3"                           │
│ 2. 从 MultiModelManager 获取 Qwen3 实例                  │
│ 3. 异步执行推理 (不阻塞其他请求)                          │
│ 4. 返回结果                                              │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│ T1: ASR Handler (与 T0 并发执行)                         │
│ 1. 解析请求,提取 model="GlmAsrNano"                      │
│ 2. 从 MultiModelManager 获取 GlmAsrNano 实例             │
│ 3. 异步执行推理 (真正的并行)                              │
│ 4. 返回结果                                              │
└─────────────────────────────────────────────────────────┘

资源共享:
✓ 两个模型共用同一个 CUDA Device
✓ Tokenizer 从缓存中直接获取（如果已加载）
✓ 内存分配器统一管理，避免碎片化
```

## 关键组件说明

### 1. SharedResources（共享资源）

```rust
pub struct SharedResources {
    pub device: Device,              // GPU/CPU 设备（只创建一次）
    pub dtype: DType,                // 数据类型（F16/F32）
    pub tokenizer_cache: RwLock<HashMap<String, Arc<Tokenizer>>>, // Tokenizer 缓存
}
```

**优势：**
- Device 初始化开销大，共享后避免重复创建
- Tokenizer 加载耗时，缓存后第二次使用瞬间完成
- 统一的内存管理策略

### 2. MultiModelManager（多模型管理器）

```rust
pub struct MultiModelManager {
    pub shared: Arc<SharedResources>,           // 共享资源引用
    pub models: RwLock<HashMap<String, Arc<ModelEntry>>>, // 模型注册表
}
```

**功能：**
- 线程安全的模型注册和查询
- 支持动态加载/卸载模型（未来扩展）
- 提供模型列表和管理接口

### 3. ModelEntry（模型条目）

```rust
pub struct ModelEntry {
    pub which_model: WhichModel,    // 模型类型标识
    pub instance: ModelInstance,    // 模型实例（包含权重、状态等）
    pub model_id: String,           // 唯一标识符
}
```

**特点：**
- 每个模型独立管理自己的 KV Cache
- 使用 `Arc` 包装，支持多线程共享读取
- 写入时使用 `RwLock` 保证安全

## 并发控制策略

### 1. 读写锁分离

```rust
// 读取模型：多个请求可以同时读取不同模型
let entry = manager.get_model("Qwen3").await;  // 读锁

// 注册模型：独占写锁
manager.register_model(id, entry).await;  // 写锁
```

### 2. 模型级别的并发

```
请求1 → Qwen3 模型 → 获取写锁 → 推理 → 释放锁
请求2 → Qwen3 模型 → 等待锁   → 推理 → 释放锁  （串行，避免 KV Cache 冲突）

请求3 → GlmAsrNano 模型 → 获取写锁 → 推理 → 释放锁  （与请求1/2 并行）
```

**原则：**
- 同一模型的多个请求串行化（保护 KV Cache）
- 不同模型的请求真正并行（最大化吞吐量）

### 3. Tokio 异步运行时

```rust
#[tokio::main]
async fn main() {
    // 所有 HTTP 请求在异步上下文中并发执行
    // 不同模型的推理任务可以并行运行
}
```

## 性能优化点

### 1. 内存优化

| 优化项 | 单模型模式 | 多模型共享模式 | 节省 |
|--------|-----------|--------------|------|
| Device 上下文 | 每次创建 | 共享 | ~50MB |
| Tokenizer | 每个模型加载 | 缓存复用 | ~10-50MB/模型 |
| 内存分配器 | 独立 | 统一 | 减少碎片化 |
| **总计** | - | - | **~15% 额外开销** |

### 2. 启动速度

```
传统方式（独立进程）:
模型1: 2s 加载 + 推理
模型2: 2s 加载 + 推理
总计: 4s + 推理时间

aha 多模型:
共享资源: 0.5s 初始化
模型1: 1.5s 加载
模型2: 1.5s 加载（并行）
总计: ~2.5s + 推理时间
```

### 3. 推理吞吐量

```
场景：10个并发请求（5个文本 + 5个语音）

独立进程:
- 需要启动2个服务实例
- 总内存: 2x 单模型占用
- 吞吐量: 受限于单个模型的并发能力

aha 多模型:
- 单个服务实例
- 总内存: 1.7x 单模型占用（共享优化）
- 吞吐量: 真正并行，接近 2x
```

## 最佳实践

### 1. 模型组合建议

✅ **推荐：**
- LLM (小) + ASR + OCR → 全能助手
- LLM (中) + Embedding → RAG 系统
- ASR + TTS → 语音对话

❌ **避免：**
- 多个大型 LLM 同时运行（内存压力大）
- 相同类型的多个模型（资源浪费）

### 2. 硬件配置

| 硬件 | 推荐配置 |
|------|---------|
| 8GB RAM | 1个小模型 + 1个ASR/OCR |
| 16GB RAM | 1个中等模型 + 2个小模型 |
| 32GB RAM | 2个中等模型 + 多个小模型 |
| GPU 8GB | 启用 F16，可运行更多模型 |
